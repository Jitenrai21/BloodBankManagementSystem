BLOOD BANK MANAGEMENT SYSTEM
Requirements & User Stories
=============================
Version: 1.0 | Date: 2026-02-20
Stack: Django 6.x | SQLite | Python 3.x

---

1. PROJECT OVERVIEW
--------------------
A web-based Blood Bank Management System to manage blood donations,
inventory, hospital requests, and fraud detection. The system supports
three user roles: Admin, Donor, and Hospital.

---

2. USER ROLES
--------------
| Role     | Description                                              |
|----------|----------------------------------------------------------|
| Admin    | Manages users, inventory, approvals, and fraud alerts    |
| Donor    | Registers, books donation slots, views history           |
| Hospital | Registers, requests blood, tracks request status         |

---

3. FUNCTIONAL REQUIREMENTS
----------------------------

3.1 Authentication & User Management
  FR-01  Users can register with email and phone.
  FR-02  Users select a role (Donor or Hospital) at registration.
  FR-03  Users can log in and log out securely.
  FR-04  Admin must approve new Hospital registrations before access is granted.
  FR-05  Admin can block/unblock any user account.
  FR-06  Passwords are stored hashed (Django bcrypt/PBKDF2).
  FR-07  Role-based access control enforced on all views.

3.2 Donor Features
  FR-08  Donors can complete a profile (name, DOB, gender, blood group, city).
  FR-09  Donors can view their donation history.
  FR-10  Donors can browse and book available donation slots.
  FR-11  Donors are flagged as ineligible if last donation was <90 days ago.
  FR-12  Donors can view their eligibility status on their dashboard.

3.3 Hospital Features
  FR-13  Hospitals can complete a profile (name, registration no., city, contact).
  FR-14  Hospitals can submit blood requests (group, units, urgency, date).
  FR-15  Hospitals can view status of all their submitted requests.
  FR-16  Hospitals can cancel pending requests.

3.4 Inventory Management
  FR-17  Admin can add blood inventory units (group, type, qty, expiry).
  FR-18  Admin can update inventory (quantity, availability).
  FR-19  System displays expiry alerts for units expiring within 7 days.
  FR-20  Inventory availability is automatically updated when requests are fulfilled.

3.5 Donation Management
  FR-21  Admin can create and manage donation slots (date, time, capacity).
  FR-22  Donors can book one active slot at a time.
  FR-23  Admin can approve or reject donation bookings.
  FR-24  On approval, inventory is updated with donated units.

3.6 Blood Request Management
  FR-25  Admin reviews and approves or rejects blood requests from hospitals.
  FR-26  System checks inventory availability before approving a request.
  FR-27  Request urgency (Normal/Urgent/Critical) is visible to admin.
  FR-28  Fulfilled requests update inventory units accordingly.

3.7 Fraud Detection
  FR-29  System checks for duplicate email/phone during registration.
  FR-30  System performs fuzzy name matching on donor records.
  FR-31  System flags donors donating more than 4 times per year.
  FR-32  System flags multiple requests originating from the same IP address.
  FR-33  Each fraud flag is assigned a risk score (0–100) and severity.
  FR-34  Admin can view, investigate, and resolve fraud flags.
  FR-35  All fraud flags are logged with timestamp, user, and IP address.

---

4. NON-FUNCTIONAL REQUIREMENTS
--------------------------------
  NFR-01  System must defend against SQL injection (Django ORM used).
  NFR-02  All forms protected with CSRF tokens.
  NFR-03  XSS prevention via Django template auto-escaping.
  NFR-04  Audit logs maintained for critical actions (approvals, blocks).
  NFR-05  System must load core pages within 2 seconds on localhost.
  NFR-06  Database: SQLite for development; migratable to PostgreSQL.
  NFR-07  Responsive UI (mobile-friendly layout).
  NFR-08  All sensitive data (passwords, tokens) must never be logged.

---

5. USER STORIES
----------------

ADMIN
  US-01  As an admin, I can log in and view a dashboard with system stats.
  US-02  As an admin, I can approve or reject hospital registrations.
  US-03  As an admin, I can block or unblock any user.
  US-04  As an admin, I can add, edit, and monitor blood inventory.
  US-05  As an admin, I can review and fulfill/reject blood requests.
  US-06  As an admin, I can create donation slots and approve bookings.
  US-07  As an admin, I can view fraud alerts with severity and risk scores.
  US-08  As an admin, I can resolve/dismiss fraud flags.

DONOR
  US-09  As a donor, I can register with my email, phone, and blood group.
  US-10  As a donor, I can complete and update my profile.
  US-11  As a donor, I can book an available donation slot.
  US-12  As a donor, I can view my full donation history.
  US-13  As a donor, I can see whether I am currently eligible to donate.

HOSPITAL
  US-14  As a hospital, I can register and await admin approval.
  US-15  As a hospital, I can submit a blood request with urgency and details.
  US-16  As a hospital, I can view and track the status of my requests.
  US-17  As a hospital, I can cancel a pending request.

6. DATA MODEL SUMMARY
-----------------------
| Model           | Key Fields                                           |
|-----------------|------------------------------------------------------|
| User            | email, phone, role, is_active, is_approved           |
| Donor           | user(FK), full_name, blood_group, last_donation_date |
| Hospital        | user(FK), name, reg_no, city, is_verified            |
| BloodInventory  | blood_group, blood_type, quantity, expiry_date       |
| DonationSlot    | date, time, max_capacity, booked_count               |
| DonationHistory | donor(FK), slot(FK), status, units_donated           |
| BloodRequest    | hospital(FK), blood_group, units, urgency, status    |
| FraudLog        | user(FK), flag_type, severity, risk_score, ip        |

---

7. TECH STACK
--------------
  Backend:    Django 6.x (Python)
  Database:   SQLite (development) → PostgreSQL (production)
  Auth:       Django built-in + custom AbstractBaseUser + PBKDF2 hashing
  Frontend:   Django Templates + HTML/CSS (Bootstrap optional)
  Fraud:      Rule-based (fuzzywuzzy, IP tracking) + optional scikit-learn
  Deployment: Render.com / Heroku (Phase 6)
